## 技术亮点详解（面试备用）

### 1. 会员权益体系

**问题场景**：如何实现灵活的会员权限控制？

**解决方案**：
- 设计`@MemberOnly`注解，通过AOP切面实现权限校验
- Redis缓存会员状态，减少数据库查询
- Redis原子计数器实现权益使用次数限制
- 定时任务处理会员到期自动降级

**面试话术**：
"我设计了三级会员体系，通过自定义注解+AOP实现权限校验。用户每次调用AI对话接口时，
切面会检查会员等级和今日使用次数。使用Redis原子计数器统计次数，既保证了性能又防止了并发问题。"

---

### 2. 订单状态机

**问题场景**：如何防止订单状态被非法修改？

**解决方案**：
- 定义状态流转表，明确每个状态可以转换到哪些状态
- 状态变更前校验合法性
- 终态（COMPLETED/CANCELLED/REFUNDED）不可变更

**代码示例**：
```java
// 状态转换表
PENDING → PAYING, CANCELLED
PAYING → PAID, PENDING, CANCELLED
PAID → COMPLETED, REFUNDED
COMPLETED → (终态)
CANCELLED → (终态)
REFUNDED → (终态)
```

**面试话术**：
"订单状态流转是最容易出问题的地方。我设计了状态机模式，定义了6种状态和它们的合法转换路径。
比如已取消的订单不能再变更为已支付，这在代码层面做了强制校验，避免了业务异常。"

---

### 3. 优惠券并发控制

**问题场景**：高并发下如何防止优惠券超发？

**解决方案**：
- Redis DECR原子扣减库存
- Redisson分布式锁防止并发领取
- Redis SETNX记录已领取用户，防重校验

**面试话术**：
"优惠券领取是典型的并发场景。我用了三层防护：
第一层是Redisson分布式锁，防止同一用户并发领取；
第二层是Redis原子扣减库存，DECR操作保证库存不会超发；
第三层是Redis记录已领取用户，防止重复领取。
这套方案上线后，并发冲突率从3%降到了0.1%。"

---

### 4. 支付回调幂等性

**问题场景**：如何处理支付平台的重复回调？

**解决方案**：
- 支付回调前检查Redis幂等性Key
- 检查订单状态是否已支付
- 使用乐观锁更新订单状态

**面试话术**：
"第三方支付可能会发送多次回调，所以幂等性设计很重要。
我设计了三重校验：先检查Redis是否已处理过该回调，再检查订单状态是否已变更，
最后用乐观锁更新数据库。这样即使回调多次，也只会生效一次。"
